## Phase 1: DBスキーマ整備（すべての土台）

### MECEチェックの結果（現状評価）

| モデル | 評価 | 問題点 |
|---|---|---|
| QuestionBank | 要改修 | ①`university`欄に「系統」「大学名」「汎用指示」が混在（意味の過積載）②公開/非公開・アーカイブの状態がない ③誰が追加したか記録がない ④面接の深掘り・難易度・出典が`modelAnswer`本文に文字列連結されている |
| PracticeRecord | 要改修 | ①どの問題バンクの問題を解いたかのリンク（`questionBankId`）がない → スコア推移・再提出・問題別分析が実装できない ②削除（非表示）フラグがない |
| StudentProfile / User | 概ね健全 | 担当講師制（生徒⇔講師の割当）が将来必要になった場合は`mentorUserId`の追加が必要（今回は対象外） |
| Task / Milestone / Document / ActivityLog | 健全 | インデックス・カスケード削除とも適切。変更不要 |
| Course / Lesson / StepAnswer / AIFeedback | 健全 | LMS系は独立しており今回のスコープ外 |
| 全体 | 注意 | 状態系がすべてString運用（enumなし）。タイポ耐性は低いが既存方針に合わせStringのまま許可リスト検証で統一する |

### 追加する列（すべてnullable or デフォルト付き = 後方互換）

```prisma
model QuestionBank {
  fieldCategory     String?  // 正規化済み系統ラベル（10系統の許可リストから）
  status            String   @default("ACTIVE") // "ACTIVE" / "PENDING"(承認待ち) / "ARCHIVED"(非表示)
  createdByUserId   String?  // 追加したユーザー（生徒投稿の承認判断・監査用）
  followUpQuestions String?  @db.Text // 面接の深掘り質問（1行1問。modelAnswerから分離）
  difficulty        String?  // 難易度（modelAnswerから分離）
  @@index([category, status])
}

model PracticeRecord {
  questionBankId String?  // 解いた問題バンクの問題ID（カスタム設問はnull）
  isArchived     Boolean  @default(false) // 生徒/講師による非表示（ソフトデリート）
  @@index([studentProfileId, questionBankId])
}
```

### 手順とリスク管理

1. Neonの**ブランチ機能でリハーサル**（本番のコピーに対して`prisma migrate deploy`を実行し検証）
2. 問題なければ本番へ適用 → その後にコード側の利用を始める（列追加のみなので稼働中アプリは無影響）
3. ロールバック: 追加列は未使用なら放置しても害がないため、コードを戻すだけで復旧できる

**承認ゲート①: 本番DBへのマイグレーション実行前に確認を取る**

## Phase 2: CSVデータ整備（NotebookLM 300問）

### 現状の問題

- 面接100問中94問・志望理由書54問・小論文82問が系統未設定 → 系統フィルタが機能しない
- `University`列に系統と大学名が混在（コード側の正規化で吸収中だが、データとして不健全）
- 面接の深掘り質問がPrompt本文に埋め込まれている（書式ゆれ4種をコードで吸収中）
- 小論文4問（NLM-024/026/028/029）に規定字数がない
- ModelAnswerに【難易度】【NotebookLM参照】【出典】が連結されている（表示時に除去中）

### 整備内容

3. 面接100問: Promptを主質問のみに整形し、深掘りを`FollowUpQuestions`へ分離
4. 全300問への系統タグ付け: AIで下書き→スプレッドシート等で**人がレビュー**→確定
5. 小論文の字数なし4問に字数を設定（または「字数指定なし」を意図として明記）
6. import script更新: 新列の検証（系統は許可リスト外を拒否・面接は一問一答チェック・字数推定率90%以上）を`test_question_bank_data.mjs`に組み込み

## Phase 3: 問題バンク管理画面 + 誰でも追加（承認制）

### 「誰が追加しても自動追加」のリスク評価と対策

| リスク | 対策 |
|---|---|
| 生徒の誤字・不完全な設問・宿題丸投げ的な設問がテナント共有バンクに混ざり品質が落ちる | **自動追加はするが`status="PENDING"`（承認待ち）で保存**し、管理画面で承認するまで他の生徒には見えない設計にする |
| 生徒が設問欄に個人情報を書いた場合、共有されると他生徒に見える | 同上（承認時に管理者が目視できる） |
| 似た設問の変種が増殖する | 同一設問文の重複防止（実装済み）+ 管理画面でのアーカイブ |
| 共通問題（NOTEBOOKLM）を編集すると再インポートで上書きされ、全テナントに影響する | 管理画面では**共通問題（tenantId=null）は編集不可・「コピーして編集」のみ**にする |
| 削除により過去の演習記録が壊れる | 物理削除はせず`status="ARCHIVED"`（選択肢から消えるだけ）。記録は設問文を自前保存しているため元々壊れない |

→ 結論: **リスクは承認フローと編集制限で潰せる。** 「生徒・講師とも実行時に自動追加（承認待ち）→ 管理者が承認/編集/アーカイブ」を推奨構成とする。

### 実装内容

1. 演習実行時（バンク外の設問）に自動で`PENDING`保存（生徒・講師とも。講師は即`ACTIVE`でも可）
2. `/settings`配下に問題バンク管理画面: 一覧（系統・種別・状態・追加者でフィルタ）/ 承認 / 編集（title・prompt・fieldCategory・modelAnswer）/ アーカイブ / コピーして編集
3. 編集・承認はメンター以上のみ（サーバー側で`assertMentor`）

## Phase 4: 演習記録の削除・再提出・スコア推移

### 「削除」「コピーを編集して再提出」の評価

- **前提の整理**: 演習記録は現在、AI採点の参照（教師データ）には使っていない（採点参照は問題バンクの模範解答のみ）。したがって「不要な教師データが省ける」効果はないが、**記録の整理**と**成長の可視化**として価値が大きい → 実装推奨
- 削除は`isArchived`によるソフトデリート（誤操作で指導履歴を失わない。メンターは表示切替で閲覧可能）
- 再提出: 記録から「この解答を編集して再提出」→ 同じ`questionBankId`で新しい記録を作成 → 同一問題の記録を時系列で束ねて**スコア推移（35点→65点→80点）を明示表示**

### 実装内容

1. 記録カードに「非表示にする」「この解答を編集して再提出」ボタン
2. 同一問題（questionBankId）の履歴をグルーピングし、記録カード上部に推移ミニグラフ（試行回数と点数）
3. 生徒ページに種別ごとのスコア推移チャート（全記録ベース）

## Phase 5: 面接の2ターン目（深掘り応答）

1. 面接の添削結果に「次の深掘りに答える」ボタン → AIが提案した深掘り質問（またはバンクの`FollowUpQuestions`）から1問を出題
2. 前ターンの主質問・回答・評価を文脈として渡し、深掘り回答を採点（会話履歴は同一セッション内で最大3ターン程度）
3. 記録は親記録にぶら下げる（`parentRecordId`列の追加が必要 → Phase 1のマイグレーションに含めるか判断）

---

## Phase 6: 週次課題割当・模範解答の後付け登録

1. **課題割当**: メンターが問題バンクから問題を選び生徒に割当 → 既存Taskとして自動作成（`type="PRACTICE"`・questionBankIdをタスクに保持）→ 生徒が提出（添削実行）で自動完了
2. **模範解答登録**: 管理画面から、カスタム/AI生成問題に模範解答・採点ポイントを登録（登録後はAI採点の参照に自動で乗る）。良かった生徒答案を「模範解答に採用」する導線も記録カードに追加（生徒名は含めない）

---

## Phase 7: 保護者向け閲覧専用リンク

1. 生徒ページの閲覧専用トークンURL（推測不能なランダムトークン・失効可能）
2. 表示内容は生徒ビューからさらに絞る（連絡先・記録の答案本文は非表示、進捗・スコア推移のみ）
3. トークンテーブルの追加が必要（`SharedAccessToken`）。**外部公開になるため設計時に必ず事前確認**
